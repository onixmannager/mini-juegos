🧱 Contratos On-Chain (Anchor)

2.1 EventFactory

Programa principal para crear eventos estándar y duelos 1v1. Emite tokens de posición como SPL tokens.

// Accounts pub struct CreateEvent { pub admin: Signer, pub event_account: Account<Event>, pub token_mint_yes: Account<Mint>, pub token_mint_no: Account<Mint>, pub vault: Account<Vault>, ... } // Event struct pub struct Event { pub admin: Pubkey, pub event_type: EventType, // Standard o Duel1v1 pub parent_event: Option, // solo duelos pub state: EventState, // Open, Active, Cutoff, Resolved, Expired pub resolution: Option, pub total_sol: u64, pub fee_basis_points: u16, // 5 a 50 (0.05% - 0.5%) pub resolution_mode: ResolutionMode, // AdminManual | SwitchboardOracle pub oracle_feed: Option, // pubkey del feed de Switchboard (si aplica) pub oracle_target_value: Option, // umbral que determina YES/NO pub oracle_condition: OracleCondition, // GreaterThan | LessThan | Equals }
🔒 Seguridad: Solo el admin designado puede resolver eventos en modo AdminManual. En modo SwitchboardOracle, la resolución la dispara cualquier keeper que presente el feed válido; el contrato verifica la firma del feed y que sea el account configurado en el evento.

2.2 Vault

Custodia SOL de cada evento. Implementa extracción del 15% en el momento del settlement.

pub fn settle_event(ctx: Context) -> Result<()> { let vault = &mut ctx.accounts.vault; let total = vault.total_sol; let protocol_share = total * 15 / 100; let winner_share = total - protocol_share; // Transferir 15% a tesorería transfer_sol(&ctx.accounts.vault, &ctx.accounts.treasury, protocol_share)?; // Distribuir resto basado en tokens ganadores distribute_to_winners(winner_share, ...)?; // Quemar tokens perdedores burn_losing_tokens(...)?; }
🔒 Protección anti-reentrada: Solana evita reentrada por diseño, pero se agrega settled: bool como flag para evitar doble cobro.

2.3 OrderBook (On-Chain)

Registra órdenes firmadas por los usuarios. El matching se realiza off-chain; la ejecución llama a una instrucción fill_order que verifica firma y cantidades.

pub fn place_order(ctx: Context, order: OrderData, signature: Vec) { // Verificar que la firma corresponde al usuario require!(signature.verify(&user.key().to_bytes(), &order.hash()), ErrorCode::InvalidSignature); // Almacenar orden con status 'open' } pub fn fill_order(ctx: Context, maker_order_id: u64, taker_order_data: OrderData, taker_sig: Vec) { // Validar firma taker, emparejar maker, transferir SOL/tokens atómicamente }
💡 Off-chain matching: El servidor escucha órdenes, encuentra coincidencias y envía la transacción fill_order a Solana. Así se ahorra gas en matching y se mantiene atomicidad on-chain.

2.4 LynxToken (SPL)

Token de gobernanza/dividendos con lógica de snapshot y burn. Implementa distribución vía Merkle tree o iteración limitada.

// Snapshot en cut-off guarda array de (holder, balance) en una cuenta comprimida pub fn snapshot_and_mint(ctx: Context, holders: Vec) { let total_supply = calculate_reward(); mint_to(&ctx.accounts.lynx_mint, &ctx.accounts.reward_vault, total_supply); for h in holders { transfer(reward_vault, h.address, h.proportional_amount); } }

📡 Off-Chain Matching Engine

Servidor que centraliza el libro de órdenes para velocidad. No custodia fondos; solo coordina transacciones.

3.1 API endpoints (protegidos con rate limiting y autenticación por wallet)

3.2 Flujo de matching

1
Usuario A firma una orden de venta (mensaje con precio, cantidad, expiración) y la envía a /order/submit.
2
El motor valida firma, nonce (para evitar replay) y la almacena en Redis/Postgres con status open.
3
Usuario B envía orden de compra compatible. El motor detecta match y llama internamente a fill_order.
4
Construye transacción Solana con las dos órdenes, verifica nuevamente firmas, y envía a la red.
5
On-chain: fill_order verifica firmas, realiza transferencia de SOL del vault al vendedor, y tokens al comprador. Ambas órdenes se marcan filled.
🔒 Protección contra front-running: Las órdenes incluyen un max_slot y un compromiso de precio. El motor rechaza órdenes con slot expirado y no revela detalles hasta el match.

⚔️ Flujo Técnico Seguro del Duelo 1v1

1
Usuario A crea duelo: Invoca create_duel en EventFactory. Elige monto X SOL, posición A. Se bloquea X SOL desde su wallet al vault del evento.
2
Evento en estado Abierto: Solo existe posición A. Nadie puede vender tokens (restricción en OrderBook).
3
Usuario B entra: Llama a enter_duel. Debe bloquear exactamente X SOL (mismo monto). Se emiten tokens de posición B y LIBRE. Estado cambia a Activo.
4
Estado Activo — trading P2P habilitado: Cualquier wallet puede comprar/vender tokens A, B o LIBRE mediante el order book. El SOL de cada trade se añade al vault.
5
Expiración sin rival: Si evento supera tiempo límite sin que B entre, función expire_duel devuelve SOL a A (menos fee de tx) y cierra.
6
Resolución (Evento padre): Cuando el evento general padre se resuelve (vía admin manual o oráculo Switchboard según el modo configurado), el duelo hereda resultado. El vault ejecuta settle automáticamente.
🔒 Atomicidad: La entrada de B es atómica: transfiere SOL y recibe tokens en una sola instrucción. Si algo falla, toda la tx se revierte.

🔮 Oráculo de Resolución de Eventos

Cada evento puede configurarse con uno de dos modos de resolución al momento de su creación. El modo es inmutable una vez activo y queda registrado on-chain en la cuenta del evento.

🛡️ Modo 1 — Admin Manual

El admin designado del evento resuelve manualmente llamando a resolve_event con el resultado (true / false). Requiere la firma de la wallet admin (idealmente multisig). Indicado para eventos subjetivos o sin feed de datos disponible.

⚡ Modo 2 — Switchboard Oracle

Se configura un oracle_feed (Pubkey de un aggregator de Switchboard) y un umbral de resolución (oracle_target_value + condición). Cualquier keeper puede disparar resolve_via_oracle cuando el feed cumple la condición. No requiere acción del admin.

5.1 Struct y Tipos

// Enum de modo de resolución pub enum ResolutionMode { AdminManual, // resolución firmada por admin SwitchboardOracle, // resolución automática vía feed } // Condición de comparación contra el valor del oráculo pub enum OracleCondition { GreaterThan, LessThan, Equals, } // Campos añadidos al struct Event pub struct Event { ... pub resolution_mode: ResolutionMode, pub oracle_feed: Option, // aggregator Switchboard pub oracle_target_value: Option, // ej: 5000_00 = $5000 con 2 dec. pub oracle_condition: OracleCondition, pub oracle_resolved_at: Option, // timestamp UNIX de resolución }

5.2 Instrucción: resolve_event (Admin Manual)

pub fn resolve_event(ctx: Context<ResolveEvent>, outcome: bool) -> Result<()> { let event = &mut ctx.accounts.event_account; // Solo permitido en modo AdminManual require!( event.resolution_mode == ResolutionMode::AdminManual, ErrorCode::WrongResolutionMode ); // Solo el admin del evento puede firmar require!( ctx.accounts.signer.key() == event.admin, ErrorCode::Unauthorized ); require!(event.state == EventState::Cutoff, ErrorCode::InvalidState); event.resolution = Some(outcome); event.state = EventState::Resolved; emit!(EventResolved { event_id: event.key(), outcome, mode: "admin" }); Ok(()) }

5.3 Instrucción: resolve_via_oracle (Switchboard)

pub fn resolve_via_oracle(ctx: Context<ResolveViaOracle>) -> Result<()> { let event = &mut ctx.accounts.event_account; // Solo permitido en modo SwitchboardOracle require!( event.resolution_mode == ResolutionMode::SwitchboardOracle, ErrorCode::WrongResolutionMode ); require!(event.state == EventState::Cutoff, ErrorCode::InvalidState); // Verificar que el account del feed es el configurado en el evento require!( ctx.accounts.switchboard_feed.key() == event.oracle_feed.unwrap(), ErrorCode::InvalidOracleFeed ); // Leer el valor del aggregator de Switchboard (via CPI o deserialización) let feed = &ctx.accounts.switchboard_feed; let feed_value: i64 = AggregatorAccountData::get_result(&feed.to_account_info()) ?.try_into() .map_err(|_| ErrorCode::OracleReadFailed)?; // Evaluar condición configurada en el evento let target = event.oracle_target_value.unwrap(); let outcome = match event.oracle_condition { OracleCondition::GreaterThan => feed_value > target, OracleCondition::LessThan => feed_value < target, OracleCondition::Equals => feed_value == target, }; event.resolution = Some(outcome); event.state = EventState::Resolved; event.oracle_resolved_at = Some(Clock::get()?.unix_timestamp()); emit!(EventResolved { event_id: event.key(), outcome, mode: "switchboard" }); Ok(()) } // Accounts necesarios para resolve_via_oracle pub struct ResolveViaOracle { pub keeper: Signer, // cualquier wallet puede ejecutar pub event_account: Account<Event>, /// CHECK: se valida manualmente que coincide con event.oracle_feed pub switchboard_feed: AccountInfo, }

5.4 Flujo completo de resolución según modo

1
Creación del evento: El admin configura resolution_mode. Si elige SwitchboardOracle, debe indicar la pubkey del feed y la condición de resolución (ej: SOL/USD > 200).
2
Estado Cutoff alcanzado: El evento cierra nuevas posiciones. A partir de aquí se puede ejecutar la resolución.
3a
[AdminManual] El admin (o multisig) firma y llama a resolve_event(outcome) directamente. El contrato valida su firma y escribe el resultado.
3b
[SwitchboardOracle] Un keeper (bot off-chain, o cualquier wallet) detecta que el feed de Switchboard cumple la condición y llama a resolve_via_oracle. El contrato lee el feed on-chain y calcula el resultado de forma trustless.
4
Settlement automático: Una vez el evento está en estado Resolved, cualquiera puede llamar a settle_event en el Vault para distribuir fondos. Emite el evento EventResolved para notificar al indexador.
🔮 Feeds de Switchboard recomendados (Mainnet): Los feeds públicos de Switchboard incluyen pares como SOL/USD, BTC/USD, ETH/USD y datos deportivos/meteorológicos a través de su red de oráculos. Consultar el directorio en app.switchboard.xyz para seleccionar el feed adecuado para cada tipo de evento.

5.5 Modo híbrido: Fallback Admin

Para mayor robustez, se puede implementar un fallback: si el oráculo no resuelve en un plazo definido (oracle_deadline), el admin recupera la capacidad de resolver manualmente como salvaguarda de último recurso.

// Añadir al struct Event: pub oracle_deadline: Option, // timestamp UNIX límite para el oráculo // En resolve_event (admin), añadir condición extra: if event.resolution_mode == ResolutionMode::SwitchboardOracle { let now = Clock::get()?.unix_timestamp; require!( now > event.oracle_deadline.unwrap_or(i64::MAX), ErrorCode::OracleDeadlineNotReached // el oráculo aún tiene tiempo ); // Si el deadline pasó y el oráculo no resolvió, el admin puede intervenir }
🔒 Seguridad del oráculo: (1) Solo se acepta el feed configurado en la creación del evento — imposible sustituirlo. (2) El valor del feed debe ser reciente (verificar latestConfirmedRound.roundOpenTimestamp dentro de un margen aceptable). (3) En modo Switchboard, el keeper no puede manipular el resultado: solo lee el estado on-chain del aggregator. (4) El fallback admin está protegido por el deadline, evitando que el admin resuelva prematuramente.

🪙 Tokens de Posición (SPL)

Cada evento crea sus propios mints (YES/NO para estándar; A/B/LIBRE para duelos). Características técnicas:

  • Decimales: 0 (cada token representa una unidad de participación, no fraccionable en este nivel).
  • Transferibilidad: Sin restricciones (cualquier wallet puede recibir/enviar).
  • Compra atómica al entrar: Al unirse a un duelo (Usuario B) o al comprar en el order book, el contrato mint_to directamente a la wallet del comprador.
  • Quema final: En el settlement, los tokens de la posición perdedora se queman con burn de SPL.
💡 ¿Por qué no fraccionables? Simplifica la lógica de pago: ganadores reciben SOL proporcional al número de tokens que poseen.

🏦 Vault y Distribución de Fondos

  • Custodia separada: Cada evento tiene su propia cuenta vault (PDA derivada del event_id).
  • 15% de comisión: En settle_event, antes de repartir, se calcula el 15% del total del vault y se transfiere a una wallet de tesorería (controlada por multisig).
  • Distribución a ganadores: El 85% restante se reparte entre los holders del token ganador. Se itera sobre una lista de holders (limitada a N para evitar límite de compute units; si son muchos, se usa un Merkle tree con reclamo posterior).
  • Reclamo pull-over-push: Para eventos masivos, los ganadores llaman a claim_winnings con prueba de Merkle, evitando bucles largos.
🔒 Protección: Solo la autoridad del programa (evento) puede iniciar el settle. El 15% no puede ser manipulado porque se calcula on-chain. Se verifica que total_sol coincida con el balance real del vault.

🪙 Lynx Token — Especificación Técnica

  • Estándar: SPL token (con metadatos Metaplex).
  • Snapshot en cut-off: Una instrucción snapshot (llamada por un keeper) registra todos los holders de tokens de posición en ese instante. Guarda un array o raíz Merkle en una cuenta.
  • Emisión: Se calcula un reward fijo por evento (governable) y se mintea proporcionalmente a los balances del snapshot.
  • Dividendos: Un pool de SOL (del 15% + fees) se distribuye periódicamente a holders de LYNX. Implementación vía Merkle distribuidor o staking.
  • Burn para acceso: Para entrar a eventos exclusivos, el usuario llama a burn_lynx que quema X LYNX y registra su acceso. El suministro se reduce, aumentando escasez.
💡 Mecanismo anti-manipulación: El cut-off es un bloque específico (o timestamp) definido al crear el evento. Nadie puede comprar tokens después sin que se refleje en el snapshot.

🛡️ Seguridad y Mitigaciones

RiesgoMitigación
ReentradaSolana no permite reentrada cross-program; se usa flag de estado settled para evitar doble ejecución.
Front-running de órdenesLas órdenes incluyen slot de expiración y se envían cifradas/commit-reveal en futuras versiones.
Manipulación del oráculoSolo se acepta el feed de Switchboard configurado en la creación del evento. El admin no puede cambiar el feed una vez activo. El keeper solo lee datos on-chain.

Admin malicioso en modo manualSe recomienda multisig (Serum Governance) para la wallet admin. Los eventos de alto valor deben usar modo oráculo.
Ataque de denegación de servicio (DoS)Rate limiting en matching engine; transacciones on-chain con prioridad de fee.
Falsificación de firmaTodas las órdenes se verifican con ed25519 contra la wallet del usuario.
Overflow/UnderflowRust y Anchor lo previenen por defecto; se usan operaciones checked.
Admin keys comprometidasMultisig (Serum Governance) para actualizaciones y resolución de eventos.
Snapshots imprecisosSe usa el estado de la blockchain en un slot exacto; no depende de fuentes externas.

🚀 Despliegue y Upgradability

Los programas de Solana son inmutables una vez desplegados, pero se puede usar un patrón proxy:

  • Programa principal (EventFactory, Vault, OrderBook): Se despliega con una clave de upgrade authority (multisig) que permite migrar a nueva lógica.
  • Estado separado: Las cuentas de datos (eventos, vaults) se almacenan en cuentas independientes, no en el código del programa, para preservar datos entre actualizaciones.
  • Matching engine: Actualizable independientemente, siempre que mantenga compatibilidad con instrucciones on-chain.
  • Integración Switchboard: La dependencia del SDK de Switchboard se fija a una versión específica. Actualizaciones del SDK requieren upgrade del programa con nueva autoridad multisig.
