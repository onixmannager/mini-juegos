🔮 Oráculo — Determinación del Resultado

El oráculo es la fuente de verdad que define al ganador. Se integra como un módulo desacoplado para permitir dos modos de operación, manteniendo la seguridad.

Modos de Resolución

ModoDescripciónConfianza
Manual (Admin) Una dirección multisig (3/5) llama a resolve_event(resultado). Válido para eventos de baja frecuencia o alta subjetividad. Social / Gobernanza
Automatizado (Switchboard) Un oráculo descentralizado (Switchboard) trae el resultado de una API verificada. El contrato consume el dato una vez que el oráculo lo publica on-chain. Criptoeconómica (stake de nodos)

Flujo Automatizado con Switchboard

1
Al crear evento estándar, el admin configura una tarea de oráculo (ej. API de resultados deportivos) y especifica el oracle_address en la cuenta del evento.
2
El evento permanece en estado Active hasta la fecha de resolución. Nadie puede resolver manualmente si el campo oracle_address no es nulo.
3
Un keeper (script off-chain) solicita al oráculo la actualización. Switchboard ejecuta la tarea, obtiene el resultado (0 o 1) y lo escribe en una cuenta de datos.
4
Inmediatamente, se invoca resolve_event on-chain, que lee el dato de la cuenta del oráculo. El contrato verifica que la cuenta pertenece al oráculo configurado y que el dato es reciente (max_age segundos).
5
Con el resultado validado, el contrato procede al settlement: calcula el 15% de comisión, distribuye el 85% entre ganadores, quema tokens perdedores. Todo en una misma transacción atómica.

Implementación en Anchor

// Instrucción de resolución con oráculo pub fn resolve_with_oracle(ctx: Context) -> Result<()> { let event = &mut ctx.accounts.event; let oracle_data = &ctx.accounts.oracle_data; // 1. Verificar que el oráculo es el designado para este evento require!(event.oracle_address == oracle_data.key(), ErrorCode::WrongOracle); // 2. Verificar que el dato es reciente (ej. menos de 60 segundos) let clock = Clock::get()?; require!(clock.unix_timestamp - oracle_data.timestamp < 60, ErrorCode::StaleData); // 3. El dato del oráculo (0 = NO, 1 = YES) define el resultado let result = oracle_data.value == 1; event.resolution = Some(result); event.state = EventState::Resolved; // 4. Llamar al vault para distribuir fondos settle_vault(ctx.accounts.vault, result)?; Ok(()) }
🔒 Validaciones críticas: Se verifica que la cuenta del oráculo coincida con la almacenada en el evento. Se impone un max_age para evitar datos obsoletos. Switchboard garantiza que solo el oráculo autorizado puede actualizar su cuenta de datos.

Oráculo para Duelos 1v1

Los duelos heredan la resolución del evento padre. Si el padre se resuelve vía oráculo, el duelo se resuelve automáticamente en la misma transacción, o el settlement del duelo se gatilla al confirmarse la resolución del padre.

💡 Flexibilidad: El diseño permite que un evento padre use oráculo y sus duelos asociados se beneficien de esa resolución descentralizada sin configuraciones adicionales.

⚔️ Flujo del Duelo con Resolución Automática

1
Usuario A crea duelo por X SOL. Se bloquea SOL en vault. Evento padre aún no resuelto.
2
Usuario B entra con X SOL. Duelo en estado Activo. Trading habilitado.
3
El evento padre está configurado con un oráculo. En la fecha de corte, el keeper invoca la actualización.
4
Switchboard publica el resultado en la cuenta on-chain. Se llama a resolve_event del padre.
5
El contrato del padre propaga la resolución automáticamente a todos los duelos hijos. El vault de cada duelo hace el settlement (15% comisión, pago a ganadores).

🛡️ Seguridad del Oráculo

  • Descentralización: Switchboard emplea una red de nodos que deben hacer staking. Manipular el resultado es costoso.
  • Múltiples fuentes: La tarea del oráculo puede configurarse para consultar 3 APIs distintas y tomar la mediana, reduciendo el riesgo de una fuente falsa.
  • Tiempo de espera (grace period): Después de que el oráculo publica, se puede configurar un retraso (ej. 60 segundos) antes de que el contrato acepte el dato, para permitir impugnaciones.
  • Modo de emergencia: Si el oráculo falla o no se actualiza en 24h, una multisig administrativa puede forzar la resolución manual como fallback.

💰 Estimación de Costos con Oráculo

Switchboard cobra por actualización (en SOL) según la complejidad. Para un escenario típico:

  • Creación de tarea (una vez): ~0.02 SOL (~$2.5 USD)
  • Por actualización (cada resolución de evento): ~0.0005 SOL (~$0.07 USD)
  • Keeper serverless (AWS Lambda): Costo mínimo (casi $0 si es bajo volumen).

Esto añade menos de $10 USD/mes al presupuesto operativo, incluso con cientos de eventos resueltos al mes.

💡 El costo de oráculo es marginal comparado con la transparencia y automatización que aporta. Para eventos de alto valor, es la opción recomendada.
